# 后台后端设计思路

## 1. 当前定位

当前后端是一个基于 FastAPI 的单体应用，承担三类职责：

- 后台管理系统 API
- 前台组学门户所需的数据查询 API
- 组学/生物信息专项能力接口

本次整理后，后端运行主体已经收敛到：

- `backend/api-server/`
- `backend/sdk/`
- `backend/mcp/`

其中：

- `main.py`：应用入口
- `register/`：应用装配层
- `core/`：全局配置、响应、安全
- `apps/`：后台业务域
- `basis/`：组学与生物信息能力域
- `db/`：数据库连接与 ORM 封装
- `libs/`、`utils/`：通用基础能力
- `conf/`：环境配置
- `static/`：Swagger 静态资源、头像等
- `alembic/`：数据库迁移
- `go-basic.obo`：组学能力依赖的本地资源文件

## 2. 当前架构的核心思路

### 2.1 分层方式

整体不是严格 DDD，也不是标准三层，而是“按业务模块切目录 + 模块内混合分层”的方式：

- 装配层：`main.py`、`register/*`
- 业务模块层：`apps/*`、`basis/*`
- 数据访问层：`*/crud.py`、`db/*`
- 基础设施层：`core/*`、`libs/*`、`utils/*`

### 2.2 两个业务大域

当前代码天然分成两个大域：

1. `apps/`
   面向后台管理与平台业务。
   主要包括认证、权限、项目、团队、样本、实验、数据资产、平台配置、北向代理、智能体接入等。

2. `basis/`
   面向组学数据和生物信息能力。
   主要包括基因组、序列、特征、表达、变异、峰、表型、种质、调控网络、生物文件解析等。

这说明当前后端其实同时承担了“管理后台后端”和“组学能力服务”两种角色。

## 3. 运行链路

### 3.1 入口链路

应用启动入口是 `backend/api-server/main.py`：

1. 创建 FastAPI 应用
2. 挂载静态资源
3. 调用 `register_*` 完成应用装配

装配顺序大致是：

1. `register_app(app)`
2. `register_router(app)`
3. `register_cors(app)`
4. `register_middleware(app)`
5. `register_exception(app)`
6. `register_init(app)`

### 3.2 路由组织

`register/router.py` 当前把两个总路由挂到统一前缀 `/api/v1` 下：

- `apps.routers.app_routers`
- `basis.routers.basis_router`

这意味着：

- 前台和后台依赖的接口统一暴露在一个 API 命名空间下
- 管理接口与组学接口没有在 API Gateway 层再做边界分割

## 4. 权限与上下文模型

### 4.1 当前权限模型

当前权限模型不是单纯用户-角色，而是带团队上下文的 RBAC：

- 系统角色：`apps/system/rbac/*`
- 团队角色：`apps/system/team/*`
- 用户与团队、角色、项目之间有关联表

`apps/services/rbd.py` 是当前权限与菜单拼装的关键服务，负责：

- 计算用户可见菜单
- 计算用户权限码
- 计算默认团队和默认项目
- 按团队返回项目列表

### 4.2 登录后的上下文

当前登录接口 `/api/v1/auth/login` 不只是返回 token，还会同时返回：

- 默认团队信息
- 默认项目信息

这决定了前端进入系统后，很多接口都默认带：

- `team_id`
- `project_id`

也就是说，这个系统的业务上下文核心不是“用户”，而是：

- 用户
- 团队
- 项目

三者组合形成当前操作上下文。

### 4.3 一个重要现实

现在大量业务接口其实隐含了“团队/项目上下文前置”的假设，例如：

- 项目列表
- 数据资产列表
- 样本与实验列表
- 动态菜单

所以未来如果要重构，不能只从“接口分模块”下手，必须把“上下文模型”先抽出来。

## 5. 数据与模块关系

### 5.1 管理域主线

从 `apps/` 看，当前主业务线大致是：

- 用户/角色/菜单/权限
- 团队/项目
- 数据资产 database
- 数据文件与元数据
- 样本 sample
- 实验 experiment
- 平台配置 platform

这条线对应的是后台管理系统。

### 5.2 组学能力主线

从 `basis/` 看，当前组学能力围绕以下对象展开：

- genome
- sequence
- feature
- expression / rnaseq
- variant
- phenome
- germplasm
- grn
- biofile

这条线更像“组学能力 API 层”，供前台查询、展示和分析使用。

### 5.3 当前耦合点

当前最强的耦合点在 `apps/databases`：

- 管理域中的“数据资产”
- 组学域中的“文件/元数据/序列能力”

是在这里被串起来的。

也就是说，现在的系统核心不是纯粹的“项目管理”，也不是纯粹的“组学查询”，而是：

- 以项目和团队为上下文
- 以数据资产为中枢
- 再把样本、实验、组学能力挂接上去

## 6. 当前架构的优点

- 业务模块已经基本按领域拆开，扩展性比单文件路由强很多。
- 管理域和组学域天然区分，后续继续拆分服务有现实基础。
- 权限模型虽然比较重，但已经能支撑团队/项目多上下文场景。
- 前端依赖的关键链路已经成型：登录 -> 用户信息 -> 菜单 -> 项目上下文 -> 业务接口。

## 7. 当前架构的主要问题

### 7.1 模块边界不够稳定

`apps/` 和 `basis/` 的边界是存在的，但不够严格：

- 有些模块是管理业务
- 有些模块是纯组学能力
- 有些模块同时承担“后台管理”和“前台查询”两种角色

这样会导致后续改一个模块时，很难判断它究竟属于后台域还是门户域。

### 7.2 代码分层不一致

不同模块内部组织方式不完全统一：

- 有的模块是 `routers + crud + schemas + models`
- 有的模块还有 `api/`
- 有的业务逻辑在 `services/`
- 有的逻辑直接写在路由函数里

这会带来两个问题：

- 新增功能时没有统一落点
- 测试与重构成本较高

### 7.3 权限依赖分散

当前权限判断分散在：

- `apps/common/depends.py`
- `apps/services/rbd.py`
- 各个接口上的 `check_permission(...)`

这导致“菜单可见性”“按钮权限”“团队上下文”“默认项目”的规则都耦合在一起。

### 7.4 配置系统偏旧且隐式

`core/config.py` + `libs.hytconfig` 当前能工作，但存在几个问题：

- 环境配置来源不够直观
- 很多路径和运行时假设是隐式的
- 缺少更清晰的配置分层

### 7.5 管理接口与组学能力接口共用一个单体

这在当前阶段是可接受的，但后续会带来几个压力：

- 发布粒度过大
- 测试范围过大
- 依赖膨胀
- 前台与后台的性能诉求混在一起

## 8. 我建议的改造方向

### 8.1 第一阶段：先稳住单体，不急着拆服务

我不建议现在立刻拆微服务。

更合理的第一阶段目标是：

- 保持单体部署
- 明确目录边界
- 统一模块分层
- 把上下文模型抽清楚

第一阶段建议按下面的思路做：

1. 固化目录边界
   - `apps/` 只保留管理后台业务
   - `basis/` 只保留组学能力与门户查询能力

2. 统一模块结构
   - 每个模块统一为 `api / service / crud / models / schemas`

3. 收敛权限模型
   - 把“用户、团队、项目上下文”抽成统一上下文对象
   - 把菜单计算、权限码计算、默认团队/项目选择集中到一个上下文服务

4. 收敛返回模型
   - 统一成功/失败响应结构
   - 降低历史接口返回风格差异

### 8.2 第二阶段：按职责拆成两个子系统

等第一阶段稳定后，可以考虑把后端逻辑概念上拆成两个子系统：

1. 管理域后端
   - 用户、角色、团队、项目、菜单、平台配置、数据资产管理

2. 组学能力域后端
   - genome / sequence / feature / rnaseq / variant / phenome / germplasm / grn / biofile

即便部署时仍然是单体，也建议代码层先按这两个边界继续收敛。

### 8.3 第三阶段：再讨论是否拆服务

只有在下面条件出现时，才值得真正拆服务：

- 门户查询量明显高于后台管理量
- 组学能力模块依赖明显更重
- 需要独立发布组学能力接口
- 需要对外开放部分能力接口

否则，过早拆服务只会增加运维与联调成本。

## 9. 我建议我们接下来优先讨论的三个问题

### 问题 1：后端未来是“一个单体两个域”，还是“后台域 / 门户域逐步拆分”

这个决定会影响：

- 目录结构
- 路由命名
- 接口边界
- 权限体系

### 问题 2：数据资产 `database` 是否继续作为核心中枢

如果继续作为中枢，那么后续设计应该围绕：

- 项目
- 数据资产
- 样本
- 实验
- 元数据

建立更清晰的关联模型。

### 问题 3：团队 / 项目上下文是否要前置成统一请求上下文

这是我认为最关键的问题。

如果这个问题不先统一，后面无论是接口规范化还是前后端联调，都会不断重复处理：

- 当前团队是谁
- 当前项目是谁
- 用户对哪些项目有权限

## 10. 我的结论

当前这套后端最合理的理解方式是：

- 它本质上是一个“带团队/项目上下文的组学数据平台单体后端”
- 后台管理只是其中一个视角
- 门户查询与组学能力是另一个视角
- 真正的业务中枢是“项目上下文下的数据资产与组学对象”

所以后续修改，我建议优先按下面顺序推进：

1. 先统一上下文模型
2. 再统一模块分层
3. 再决定后台域与组学域的代码边界
4. 最后再决定要不要拆服务
