# 种质资源动态字段设计草案

## 1. 这份文档解决什么问题

本文档专门回答种质资源导入中的一个核心设计问题：

- 固定字段没有问题，应该作为标准模板的一部分长期保留
- 固定字段之后的列，不应该再被理解为“某个物种固定模板”
- 这些扩展列在现实中可能每次提交都不同
  - 即使是同一个物种
  - 即使来自相近的数据来源
  - 即使由不同人整理

因此，种质资源模块不能建立在“所有扩展字段都要先被模板预定义”的假设上。

这份文档的目标是把种质资源字段体系收束成一个更贴近真实数据的三层设计：

1. 固定字段
2. 批次字段快照
3. 行级动态字段值

---

## 2. 设计结论

### 2.1 固定字段保留

固定字段继续保留，而且要明确放入模板中，让用户知道哪些列名必须按系统约定填写。

当前建议固定字段包括：

- `ID`
- `chinese_name`
- `english_name`
- `P`
- `M`
- `P_name`
- `M_name`

这些字段的特点是：

- 跨物种都能解释
- 和种质身份、显示名称、谱系关系直接相关
- 应该稳定落入主表

### 2.2 不做“扩展字段预定义模板”

对于固定字段之后的扩展列，不建议在当前阶段做“模板定义层强约束”。

原因不是模板没有价值，而是它解决不了你当前的真实问题：

- 扩展字段不一定按物种稳定
- 同一物种不同批次字段也可能不同
- 用户未必愿意在导入前先注册一套字段体系

因此当前阶段的正确假设应该是：

- 每次导入都可能带来一组新的扩展字段
- 系统要接受这种不一致
- 但系统必须把这次导入的字段结构记录下来

### 2.3 动态字段的正确核心不是“全局模板”，而是“批次字段快照”

动态字段真正需要的不是一个全局模板定义表，而是：

- 系统能够知道“这一次导入到底有哪些扩展字段”
- 系统能够知道“这些字段在这一批数据里如何解释和展示”

所以，当前阶段最关键的新增对象不是“模板定义表”，而是“批次字段快照”。

---

## 3. 为什么只靠 `attributes_json` 不够

当前实现里，固定字段之外的列会被放入 `attributes_json`。这个方向本身没有问题，但如果只有它，还不够。

如果只存：

- 固定字段
- `attributes_json`

会出现下面几个问题：

1. 系统不知道这一批数据有哪些动态字段
2. 系统不知道这些字段的原始表头是什么
3. 系统不知道动态字段的显示顺序
4. 列表页不知道该展示哪些扩展列
5. 批次 A 和批次 B 字段不一致时，系统无法正确解释

举例：

批次 A 的扩展字段可能是：

- `育种历史`
- `花朵性状`
- `植株特征`
- `用途`

批次 B 的扩展字段可能是：

- `育种记录`
- `花色`
- `花型`
- `应用方向`

如果这两批都只往 `attributes_json` 塞值，数据库虽然能存住，但系统并不知道：

- 这些字段是哪个批次出现的
- 页面应该按什么顺序显示
- 两批数据能不能放在同一个表里展示

所以，`attributes_json` 负责存值，但还需要一个对象来描述“这批值的字段结构”。

---

## 4. 推荐的三层设计

## 4.1 第一层：固定字段

这一层继续放在主表 `brd_germplasm`。

建议保留为结构化列的字段：

- `taxonomy_tax_id`
- `accession_id`
- `display_name`
- `english_name`
- `father_accession`
- `mother_accession`
- `father_name_snapshot`
- `mother_name_snapshot`
- `status`
- `source_row_no`

这层解决的是：

- 系统必须认识哪些字段
- 哪些字段可以稳定检索
- 哪些字段可以支持谱系关系构建

## 4.2 第二层：批次字段快照

这一层建议放在 `brd_germplasm_import_batch`。

建议新增字段：

- `field_schema_json`

作用不是定义全平台统一模板，而是冻结这次导入的动态字段结构。

建议其中至少记录：

- `field_key`
- `field_label`
- `source_header`
- `display_order`
- `data_type`
- `is_builtin`
- `is_dynamic`

建议示例：

```json
[
  {
    "field_key": "breeding_history",
    "field_label": "育种历史",
    "source_header": "育种历史",
    "display_order": 1,
    "data_type": "string",
    "is_builtin": 0,
    "is_dynamic": 1
  },
  {
    "field_key": "flower_traits",
    "field_label": "花朵性状",
    "source_header": "花朵性状",
    "display_order": 2,
    "data_type": "string",
    "is_builtin": 0,
    "is_dynamic": 1
  }
]
```

这一层解决的是：

- 这批数据到底有哪些扩展字段
- 原始 Excel 表头是什么
- 页面按什么顺序展示
- 导出时按什么顺序恢复字段
- 同物种不同批次字段不一致时，如何保持每批数据自解释

## 4.3 第三层：行级动态字段值

这一层继续放在 `brd_germplasm.attributes_json`。

建议仍然存成键值结构：

```json
{
  "breeding_history": "示例历史1",
  "flower_traits": "蓝紫色重瓣",
  "usage": "切花"
}
```

这里的重点是：

- `attributes_json` 只负责存值
- 字段怎么展示、怎么解释，不由它单独承担
- 解释权来自所属批次的 `field_schema_json`

---

## 5. `field_schema_json` 到底怎么解决“不一致”

这里最容易误解。

`field_schema_json` 不是为了让所有批次保持一致。

它解决的是另一个问题：

- 既然现实中每次导入都可能不一样，那系统至少要保留“这一次当时是什么样子”

也就是说，它是“批次自解释机制”，不是“全局统一机制”。

### 5.1 它解决的问题

当用户打开某个导入批次时，系统可以准确知道：

- 这批数据有哪些动态列
- 这些列当时叫什么名字
- 页面该按什么顺序显示
- 详情页应该如何解释 `attributes_json`

### 5.2 它不解决的问题

它并不试图强行解决：

- 批次 A 和批次 B 字段完全统一
- 不同提交者使用同一个字段名称体系
- 跨批次字段自动标准化

这些属于后续更高阶段的“字段标准化”问题，不应该在当前阶段强行引入。

### 5.3 为什么它是必须的

如果没有这层，系统面对动态字段时只能看到一团 JSON 值。

有了这层，系统就能做到：

- 单批次列表页动态列展示
- 单批次导出恢复原字段结构
- 详情页字段分组展示
- 后续再做字段标准化时，有历史依据可追溯

---

## 6. `field_key` 应该怎么设计

这里建议不要直接把原始 Excel 表头当作最终存储键。

更稳妥的做法是：

- `source_header` 保留用户原始表头
- `field_label` 保留当前展示名称
- `field_key` 使用系统生成或规范化后的稳定键

例如：

| source_header | field_label | field_key |
|---|---|---|
| `育种历史` | `育种历史` | `breeding_history` |
| `花朵性状` | `花朵性状` | `flower_traits` |
| `用途` | `用途` | `usage` |

如果系统暂时无法自动把中文可靠地转换成稳定键，也可以在第一阶段使用：

- `attr_001`
- `attr_002`
- `attr_003`

然后通过 `field_schema_json` 保存映射关系。

这两种方案里，我更推荐：

- 优先做“可读型规范键”
- 转换失败时回退成 `attr_xxx`

原因：

- 程序内部更稳定
- 后续检索和导出更清晰
- 不依赖原始表头永远不变

## 7. 导入流程如何调整

## 7.1 当前建议流程

导入时按下面的顺序处理：

1. 读取 Excel
2. 跳过 `#` 注释行
3. 识别固定字段表头
4. 剩余列全部视为本次批次的动态字段
5. 为这些动态字段生成 `field_schema_json`
6. 每一行的动态值写入 `attributes_json`
7. 固定字段继续写入 `brd_germplasm` 主表

### 7.2 动态字段校验建议

动态字段不要求预注册，但建议仍做基础校验：

- 列名不能为空
- 动态列名不能重复
- 动态列名不能和固定字段冲突
- 列数过大时给出提示

### 7.3 导入结果建议返回

`validate` 接口除了现有结果外，建议额外返回：

- `builtin_fields`
- `dynamic_fields`
- `field_schema_preview`

这样前端在“预览导入结果”阶段就能告诉用户：

- 系统识别到哪些固定字段
- 这次会新增哪些动态字段

## 8. 前端页面如何使用这套设计

## 8.1 导入页

导入页只需要提供：

- 固定字段模板下载
- 固定字段帮助说明
- 本次识别到的动态字段预览

不需要要求用户先定义一套动态模板。

## 8.2 种质列表页

列表页建议分两种展示模式：

### A. 按批次浏览

最适合当前阶段。

特点：

- 先选物种
- 再选批次
- 动态列按该批次的 `field_schema_json` 展示

这样最容易避免字段混杂。

### B. 跨批次浏览

这可以做，但建议作为后续增强。

因为一旦跨批次，不同批次的动态字段很可能不一致，需要做：

- 动态列并集
- 空值填充
- 同义字段合并策略

这不适合当前第一阶段强行做重。

## 8.3 详情页

详情页建议拆成三块：

- 基础信息
- 谱系信息
- 动态属性

其中“动态属性”区域应按该记录所属批次的 `field_schema_json` 来渲染字段名称和顺序。

---

## 9. 数据库调整建议

在当前设计下，数据库只需要做一个很小但关键的增强。

## 9.1 保留现有表

继续保留：

- `brd_germplasm_import_batch`
- `brd_germplasm`
- `brd_germplasm_lineage`

## 9.2 在 `brd_germplasm_import_batch` 中新增字段

建议新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| `field_schema_json` | text | 本批次动态字段结构快照 |

可选新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| `builtin_field_snapshot_json` | text | 本批次固定字段识别快照 |
| `template_version` | varchar(64) | 固定字段模板版本，如 `crop_germplasm_v1` |

## 9.3 `brd_germplasm.attributes_json` 继续保留

当前阶段不建议拆表。

原因：

- 实现成本低
- 足够支持详情页和单批次浏览
- 也便于后续观察真实用户到底会提交哪些字段

等后面真的需要做：

- 动态字段筛选
- 动态字段聚合统计
- 跨批次字段检索

再考虑拆成单独的值表。

---

## 10. 后续扩展路线

当前阶段不做全局字段注册表，但未来如果真的出现这些需求：

- 跨批次统计哪些动态字段最常出现
- 把 `花朵性状` 和 `花型描述` 标准化为同一字段
- 支持按动态字段做高级筛选

那时可以再补：

- `brd_germplasm_field_registry`
- `brd_germplasm_field_alias`
- `brd_germplasm_attr_value`

但这些都应该建立在“先收集真实批次字段”的基础上，而不是在一开始凭空定义。

---

## 11. 最终结论

种质资源的字段体系不应该设计成：

- 固定字段 + 物种级固定扩展模板

更合理的设计是：

- 固定字段模板
- 批次字段快照
- 行级动态字段值

一句话收束：

- 固定字段解决“系统必须认识什么”
- `field_schema_json` 解决“这次导入到底有哪些动态字段”
- `attributes_json` 解决“每条记录这些动态字段的值是什么”

这比“先定义好所有模板”更贴近你现在的真实数据模式，也更适合社区版先把数据落下来。
