# 育种实验设计域与组学数据资产整合设计草案

## 1. 文档目标

本文档用于定义当前 Dataset 平台与未来育种业务域之间的整合方式。

这份文档不再把 `project / sample / experiment / files` 直接作为平台总模型，而是明确分成两层：

- `Dataset` 资产域
- `Breeding` 业务域

其中：

- `Dataset` 资产域负责文件、版本、资产、权限、发布和查询
- `Breeding` 业务域负责 `Program / Material / Trial / Plot / Observation / BioSample / Assay`

最重要的设计约束是：

**两层必须解耦，且必须支持延迟绑定。**

也就是说：

- 用户可以先把基因组、表达、变异等数据文件注册到平台
- 即使没有任何育种元数据，这些 dataset 也必须能正常落库、管理和查询
- 等未来有了 `Program / Material / Trial / BioSample / Assay` 等结构化元数据，再与 dataset 做关联

本文档用于把这条原则正式化，并为后续持续打磨提供统一底稿。

## 3. 当前问题重新定义

之前的讨论里，一个容易走偏的方向是：把“项目、样本、实验”当成所有组学数据接入的统一入口。

这在 NCBI SRA 或传统组学仓库里是合理的，因为它们主要关心：

- 项目
- 样本
- 实验
- 文件

但当前平台并不是一个纯 SRA 仓库，而是一个以文件资产管理为核心、并希望后续兼容育种业务的平台。

因此当前真正的问题不是：

- “如何让所有数据都先落到 `project/sample/experiment`”

而是：

- “如何让文件资产域先独立成立，并允许后续与育种业务域做结构化关联”

这是本次修正后的设计核心。

## 4. 设计目标

本设计需要同时满足以下目标：

1. 文件资产先落库
   - 用户只有文件，没有任何结构化业务元数据时，也能把数据接入平台

2. 业务元数据后补录
   - 后续可逐步建设 `Program / Material / Trial / Plot / Observation / BioSample / Assay`

3. 资产与业务可关联
   - 当育种或组学元数据准备好后，能把 dataset 精确绑定到材料、样本、实验或试验对象

4. 不强制所有组学都走同一条业务链
   - 基因组数据通常不依赖育种业务域
   - RNA-Seq 和 Variant 常依赖 `BioSample / Assay`
   - Phenotype 常依赖 `Trial / Plot / Observation`

5. 保持当前 Dataset 平台主线稳定
   - 不为了补育种模型，反向破坏 `dataset/version/asset/file/lineage` 当前已形成的主线

## 5. 设计原则

### 5.1 两域解耦

系统明确分成两个业务域：

#### A. Dataset 资产域

负责：

- 文件注册
- 版本管理
- 资产管理
- 发布语义
- 权限语义
- 查询路由
- lineage

#### B. Breeding 业务域

负责：

- `Program`
- `Material`
- `Trial`
- `Plot`
- `Observation`
- `BioSample`
- `Assay`

这两个域不应该互相卡死。

### 5.2 先资产化，再结构化，再关联化

建议平台长期遵循以下顺序：

1. 先资产化
   - 先让文件进入 Dataset 平台

2. 再结构化
   - 再把育种和组学业务对象整理为结构化元数据

3. 最后关联化
   - 再把 dataset 与 `Material / Plot / BioSample / Assay` 建立正式关系

### 5.3 延迟绑定

平台必须允许：

- dataset 在创建时没有 `Program`
- dataset 在创建时没有 `Material`
- dataset 在创建时没有 `BioSample`
- dataset 在创建时没有 `Assay`

这些关系都允许后补。

### 5.4 以育种主线解释育种业务

一旦进入育种域，系统应优先遵循 breeding 文档里的主线，而不是 SRA 逻辑。

育种主线应是：

`Program -> Material -> Trial -> Plot -> Observation -> SelectionDecision`

组学增强主线应是：

`Material -> BioSample -> Assay -> DataFile`

而不是用 `Sample` 取代 `Material` 作为一级主体。

### 5.5 SRA 模型仅作为组学辅线，不作为平台总模型

当前已有的：

- `project`
- `sample`
- `experiment`

可以继续保留和兼容，但它们不应再被当作平台顶层真相模型。

更准确地说：

- 它们更接近组学管理中的旧式或 SRA 风格对象
- 后续应逐步向 `Program / BioSample / Assay` 语义收敛

## 6. 两域总体结构

### 6.1 Dataset 资产域

当前已存在并继续沿用：

- `dataset`
- `dataset_version`
- `dataset_asset`
- `asset_file`
- `dataset_lineage`

这是平台底层事实域。

它回答的问题是：

- 这是什么数据文件
- 有哪些版本
- 每个版本有哪些资产
- 每个资产有哪些主文件和索引文件
- 当前谁能看、谁能查、哪个版本是公开版本

### 6.2 Breeding 业务域

后续建设并逐步落地：

- `program`
- `material`
- `trial`
- `trial_entry`
- `plot`
- `plot_plant`
- `trait`
- `protocol`
- `observation_batch`
- `observation`
- `biosample`
- `assay`

这是平台结构化业务解释层。

它回答的问题是：

- 这个材料属于哪个育种项目
- 这个材料参加过哪些试验
- 哪个 plot 上发生了哪些观测
- 哪个生物样本做了哪些 assay
- 哪个 assay 产出了哪些组学结果

### 6.3 关联层

位于两域之间的薄层，用于实现后补关联。

建议新增：

- `dataset_subject_link`
- `dataset_assay_link`
- `variant_sample_map`
- `phenotype_subject_map`

这层回答的问题是：

- 这个 dataset 与哪个主体有关
- 这个 dataset 是哪个 assay 的产物
- 这个 VCF 的 sample 列对应哪个材料或样本
- 这个 phenotype dataset 中的一行记录对应哪个 plot 或 material

---

## 7. Breeding 域采用的主对象

本设计在业务语义上，正式采用 breeding 项目中的对象体系。

### 7.1 Program

`Program` 是育种项目级容器。

它对应 breeding 文档中的：

- 项目
- 物种
- 年度主线

注意：

- `Program` 不是 dataset 注册前置条件
- 它只在用户进入育种业务管理时才成为主对象

### 7.2 Material

`Material` 是一级主数据。

它是育种平台中最重要的主体，用于承载：

- 品种
- 品系
- 群体成员
- 选系材料
- 杂交后代

这是后续所有跨组学整合的首选对齐键。

### 7.3 Trial / Plot / Observation

这是育种业务主链。

主线是：

`Material -> TrialEntry -> Plot -> Observation`

说明：

- `Trial` 表示试验
- `Plot` 表示材料在试验中的最小执行单元
- `Observation` 表示表型或其它试验观测值

多数表型组数据最终都应归属到这条链路，而不是直接归属到文件。

### 7.4 BioSample

`BioSample` 是组学取样对象。

它应与下列对象之一或多者有关：

- `Material`
- `Plot`
- `PlotPlant`

它不是育种平台的一号主对象，但它是组学接入育种平台时的关键桥梁。

### 7.5 Assay

`Assay` 是组学实验事件。

例如：

- WGS
- RNA-seq
- Genotyping
- Metabolome

它与 `BioSample` 相连，并进一步产出组学文件或结果。

## 8. Dataset 域继续独立成立

这是当前设计里必须写清的一条规则。

### 8.1 可独立接入的数据类型

以下类型的数据应允许在没有育种域元数据的情况下，直接注册到 Dataset 平台：

- 参考基因组
- 基因注释
- 功能注释
- 表达矩阵
- 变异 VCF/BCF
- 信号轨道
- 互作矩阵
- phenotype 原始表文件

### 8.2 接入时不要求业务对象

Dataset 注册时不要求以下字段：

- `program_id`
- `material_id`
- `trial_id`
- `plot_id`
- `biosample_id`
- `assay_id`

如果用户有这些元数据，可以后续补充；
如果没有，dataset 也必须能先注册成功。

### 8.3 这条原则的实际意义

这意味着：

- 一个基因组目录可以独立注册
- 一个 transcriptome HDF5 可以独立注册
- 一个 VCF 可以独立注册
- 一个 phenotype xls 也可以先独立注册

平台不会因为“缺少业务上下文”而拒绝资产落库。

## 9. 关联层的正式定位

关联层不是新的主业务域，而是 Dataset 域与 Breeding 域之间的桥。

这一层的设计目标是：

1. 不阻塞 dataset 先注册
2. 支持后补录
3. 支持部分完成映射
4. 区分“自动推断”与“人工确认”
5. 支持多主体、多版本、多资产并存

建议关联层统一遵循以下公共字段风格：

- `mapping_status`
- `mapping_method`
- `confidence`
- `is_primary`
- `note`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

推荐统一枚举：

- `mapping_status`
  - `draft`
  - `matched`
  - `reviewed`
  - `rejected`
- `mapping_method`
  - `manual`
  - `rule`
  - `import`
  - `inferred`
- `confidence`
  - `low`
  - `medium`
  - `high`

### 9.1 关联粒度规则

建议统一采用以下粒度规则：

#### A. 默认优先挂到 `asset`

如果关系能够明确到具体资产，优先记录：

- `dataset_id`
- `version_id`
- `asset_id`

原因：

- 同一个 version 下可能有多个业务资产
- 每个资产对应的业务含义可能不同

例如：

- 一个 transcriptome version 下既有 `expression_matrix`
- 也有 `sample_metadata`
- 它们不一定对应完全相同的业务主体

#### B. `asset_id` 可空，但必须有清晰语义

如果关系本身是版本级关系，可以允许：

- `asset_id = null`

例如：

- 某个 version 整体属于某个 Program
- 某个 version 整体对应某次导入成果

#### C. 不建议只挂 `dataset_id`

仅挂 `dataset_id` 会丢失版本语义。

因此建议：

- 最少挂到 `version_id`
- 尽量挂到 `asset_id`

### 9.2 dataset_subject_link

用于把 dataset/asset 关联到具体业务主体。

这一层在逻辑上依然叫 `subject link`，但数据库落地不建议再使用纯文本 `subject_type + subject_id` 组合。

原因：

- PostgreSQL 无法对这种“多态外键”提供真正的外键完整性
- 后续一旦接前端筛选、统计、删除策略，维护成本会迅速上升
- 当前平台已经决定走 PostgreSQL 主库，应该尽量利用真实 FK 和检查约束

因此本设计在数据库实现上建议改成：

- `program_id`
- `material_id`
- `trial_id`
- `plot_id`
- `biosample_id`

这些列允许为空，但要求：

- 必须且只能有一列非空
- 由检查约束保证一致性
- `role` 继续表达“这份 dataset 与主体之间是什么关系”

建议字段：

- `id`
- `dataset_id`
- `version_id`
- `asset_id`
- `program_id`
- `material_id`
- `trial_id`
- `plot_id`
- `biosample_id`
- `role`
- `mapping_status`
- `mapping_method`
- `confidence`
- `is_primary`
- `note`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

建议 `role` 可包括：

- `owned_by`
- `about`
- `sampled_from`
- `generated_for`
- `used_in`
- `supports_decision_for`

建议唯一性规则：

- 当 `asset_id` 非空时：
  - `(asset_id, role, program_id, material_id, trial_id, plot_id, biosample_id)` 唯一
- 当 `asset_id` 为空时：
  - `(version_id, role, program_id, material_id, trial_id, plot_id, biosample_id)` 唯一

这张表的意义是：

- 一个 dataset 可以先注册
- 之后再补“它与哪个 subject 有关”

推荐用途示例：

1. 基因组 dataset 与 Program 的关系
   - `program_id = ...`
   - `role = used_in`

2. 某 phenotype 表与 Trial 的关系
   - `trial_id = ...`
   - `role = about`

3. 某表达矩阵与 BioSample 的关系
   - `biosample_id = ...`
   - `role = generated_for`

### 9.3 dataset_assay_link

用于把 dataset/asset 关联到 assay。

建议字段：

- `id`
- `dataset_id`
- `version_id`
- `asset_id`
- `assay_id`
- `role`
- `mapping_status`
- `mapping_method`
- `confidence`
- `is_primary`
- `note`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

典型 `role`：

- `raw_reads`
- `clean_reads`
- `alignment`
- `quantification`
- `variant_calls`
- `phenotype_measurement_table`
- `analysis_result`

建议唯一性规则：

- `(asset_id, assay_id, role)` 唯一

这张表的定位比 `dataset_subject_link` 更强：

- `dataset_subject_link` 表达“和哪个主体有关”
- `dataset_assay_link` 表达“是哪个 assay 的产物或中间结果”

推荐用途示例：

1. RNA-Seq assay 产出表达矩阵
   - `role = quantification`

2. DNA assay 产出 VCF
   - `role = variant_calls`

3. 田间表型 assay 产出 phenotype 清洗表
   - `role = phenotype_measurement_table`

### 9.4 variant_sample_map

这是 Variant 组学与育种域连接的关键表。

VCF sample 列不应只保留为文本，而应允许后续映射到：

- `material`
- `biosample`
- `plot`

建议字段：

- `id`
- `dataset_id`
- `version_id`
- `asset_id`
- `vcf_sample_name`
- `sample_alias`
- `normalized_sample_name`
- `material_id`
- `biosample_id`
- `plot_id`
- `mapping_status`
- `mapping_method`
- `confidence`
- `is_primary`
- `note`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

建议唯一性规则：

- `(asset_id, vcf_sample_name)` 唯一

建议映射优先级：

1. `material`
2. `biosample`
3. `plot`

注意：

- 这也是延迟绑定模型
- VCF 可以先注册
- `variant_sample_map` 可以后补

建议产品层支持以下完成度展示：

- `unmapped`
- `partially_mapped`
- `fully_mapped`
- `review_required`

推荐用途示例：

- 一个 VCF 有 320 个 sample 列
- 其中 300 个已映射到 `material`
- 15 个映射到 `biosample`
- 5 个仍待人工确认

### 9.5 phenotype_subject_map

表型数据大多数不会一开始就完全结构化，因此建议为过渡期保留一层 phenotype 数据与主体的映射表。

建议字段：

- `id`
- `dataset_id`
- `version_id`
- `asset_id`
- `row_key`
- `trial_id`
- `plot_id`
- `material_id`
- `trait_code`
- `obs_date`
- `mapping_status`
- `mapping_method`
- `confidence`
- `is_primary`
- `note`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

其中 `row_key` 建议语义为：

- 原始文件中的稳定行标识
- 或导入时生成的行 hash
- 不应依赖页面排序号

建议唯一性规则：

- `(asset_id, row_key)` 唯一

建议映射优先级：

1. `plot`
2. `material`
3. `trial`

这有助于 phenotype 在“文件表格阶段”就逐步进入业务体系。

推荐用途示例：

- 一个 phenotype xls 的第 245 行
- `row_key = 2026HN001_sheet1_row245`
- 当前已映射到：
  - `trial_id = 18`
  - `plot_id = 2045`
  - `trait_code = YLD`
  - `obs_date = 2026-10-12`

### 9.6 关联层完成度语义

建议所有关联层对象统一支持“完成度”概念，而不是只有“有没有映射”。

推荐统一口径：

- `unlinked`
  - 完全没有建立关系
- `partially_linked`
  - 已有部分关系，但不完整
- `fully_linked`
  - 当前要求范围内关系完整
- `review_required`
  - 自动匹配完成，但仍需人工确认

不同数据类型的“完整”标准允许不同：

- `Genome`
  - 允许长期 `unlinked`
- `Transcriptome`
  - 至少有 reference lineage
  - 更高标准是再补 `biosample/assay`
- `Variant`
  - 至少有 reference lineage
  - 更高标准是 `variant_sample_map` 完整
- `Phenotype`
  - 至少先有 dataset 资产
  - 更高标准是进入 `trial/plot/observation`

## 10. 各数据类型接入策略

接入策略统一分成三种动作：

1. 资产注册
2. 结构化解释
3. 业务关联

其中：

- 第 1 步必须始终可独立完成
- 第 2、3 步允许滞后

### 10.0 统一接入模板

建议平台内部统一按以下模板理解任何数据接入：

#### Step A：Register

把文件接入 Dataset 平台：

- 创建 dataset
- 创建 version
- 创建 assets
- 登记 files
- 建立 lineage

#### Step B：Interpret

从文件中抽取更稳定的结构化对象：

- phenotype 抽取 trait / row / sheet 信息
- variant 抽取 VCF sample 名
- expression 抽取 sample 列和 matrix 类型

#### Step C：Link

把解释出的对象挂到 breeding 域：

- 挂 program
- 挂 material
- 挂 plot
- 挂 biosample
- 挂 assay

这三步应支持分开执行。

### 10.1 Genome

基因组数据完全可以先只存在于 Dataset 域：

`sequence dataset -> version -> assets`

它通常不需要 `Program / Material / Trial / BioSample / Assay`。

如果未来 breeding 域需要声明某 Program 使用哪个参考基因组，可以通过：

- `dataset_subject_link(program_id = ...)`

后补。

推荐完成标准：

- 最低标准：
  - sequence dataset 已注册
- 增强标准：
  - 已绑定 program 或 species 语义

### 10.2 Transcriptome / RNA-Seq

推荐三阶段：

### 阶段 A：文件资产先注册

- 注册为 `dataset_type = expression`
- 绑定参考基因组 lineage
- 支持查询、展示、发布

### 阶段 B：结构化解释

- 识别样本列
- 识别 matrix 类型
- 识别 sample metadata

### 阶段 C：业务关系后补

后续再追加：

- `dataset_assay_link`
- `dataset_subject_link(biosample_id = ...)`

这意味着 RNA-Seq 不应因为没有 `BioSample / Assay` 而无法进入平台。

推荐完成标准：

- 最低标准：
  - dataset 已注册
  - 已绑定 reference lineage
- 中间标准：
  - matrix/sample metadata 已结构化
- 完整标准：
  - 已绑定 `biosample`
  - 已绑定 `assay`

### 10.3 Variant / VCF

同样采用三阶段：

### 阶段 A：先资产化

- 注册 `variant dataset`
- 完成索引、查询、版本、发布

### 阶段 B：结构化解释

- 提取 sample 列
- 形成待映射 sample 清单

### 阶段 C：后映射

- 补 `variant_sample_map`
- 再与 `material / biosample / plot` 对齐

这是 Variant 正确的接入方式。

推荐完成标准：

- 最低标准：
  - dataset 已注册
  - 已绑定 reference lineage
- 中间标准：
  - sample 列已抽取
- 完整标准：
  - `variant_sample_map` 已达到可分析要求

### 10.4 Phenotype / XLS

Phenotype 建议也先允许文件资产化，而不是要求一开始就把所有 trial/plot/trait 都准备好。

推荐分三步：

1. 原始 phenotype xls/csv 先注册为 dataset
2. 后续做结构化解析，得到 phenotype 结构化表
3. 再通过 `phenotype_subject_map` 或正式 observation 表挂到 `Trial / Plot / Material`

因此 phenotype 也应先允许进入 Dataset 平台，而不是被业务建模卡住。

推荐完成标准：

- 最低标准：
  - phenotype 文件已注册为 dataset
- 中间标准：
  - 行、列、trait 字段已解释
- 完整标准：
  - 已进入 `trial / plot / observation` 主线

### 10.5 联合分析前的最低绑定要求

如果后续要支持联合分析，建议最低绑定要求如下：

#### Expression × Genome

- 必须有 reference lineage

#### Variant × Genome

- 必须有 reference lineage

#### Variant × Phenotype

- Variant 侧至少有 `variant_sample_map`
- Phenotype 侧至少能映射到 `material` 或 `plot`

#### GWAS

- Variant 必须能对齐到 `material`
- Phenotype 必须能回到 `material`

也就是说，GWAS 真正的前提不是“文件都存在”，而是“两个域之间已经建立足够稳定的桥”。

## 11. 数据库表草案

本节把上一节的逻辑模型压成第一版 PostgreSQL 表草案。

配套 SQL 草案见：

- `docs/育种实验设计域数据库DDL草案.sql`

这里的目标不是一步到位，而是：

- 先把 `Program / Material / Trial / Plot / Observation / BioSample / Assay / DataFile` 这条主线落成可建表的草案
- 再把 4 张关联表落成可执行数据库设计
- 同时保持与当前 Dataset 平台解耦

### 11.1 建模约定

推荐约定如下：

- 所有 breeding 域表统一加前缀 `brd_`
- 主键统一用 `bigserial`
- 时间字段统一用 `timestamptz`
- 扩展字段统一优先用 `jsonb`
- 逻辑删除优先，不建议直接级联物理删除
- 审计字段统一保留：
  - `created_at`
  - `updated_at`
  - `created_by`
  - `updated_by`

补充建议：

- 所有状态字段统一用 `varchar(32)`
- 所有编码类字段统一使用唯一索引
- 所有外键优先 `on delete restrict` 或 `on delete set null`

说明：

- 为了先满足你当前要求，本轮只落 8 张核心表
- `trial_entry / trait / protocol / plot_plant / block / treatment` 暂不在本轮展开
- 其中 `plot` 表先直接持有 `material_id`，后续如需要更完整试验设计，再拆出 `trial_entry`

- `material_id/biosample_id/plot_id` 允许为空
- 至少允许“先抽样本名、后补映射”

索引建议：

- `unique(asset_id, vcf_sample_name)`
- `index(material_id)`
- `index(biosample_id)`
- `index(mapping_status)`

### 11.13 `brd_phenotype_subject_map`

用途：

- 记录 phenotype 文件中每一行与 trial/plot/material 的映射
- 作为 phenotype 从文件阶段过渡到 `Observation` 阶段的桥

建议字段：

| 字段 | 类型 | 空值 | 说明 |
|---|---|---|---|
| id | bigserial | 否 | 主键 |
| dataset_id | bigint | 否 | dataset |
| version_id | bigint | 否 | version |
| asset_id | bigint | 否 | asset |
| row_key | varchar(128) | 否 | 行稳定标识 |
| trial_id | bigint | 是 | Trial，可空 |
| plot_id | bigint | 是 | Plot，可空 |
| material_id | bigint | 是 | Material，可空 |
| trait_code | varchar(32) | 是 | 性状编码 |
| obs_date | date | 是 | 观测日期 |
| mapping_status | varchar(32) | 否 | 映射状态 |
| mapping_method | varchar(32) | 否 | 映射方式 |
| confidence | varchar(16) | 是 | 置信度 |
| is_primary | boolean | 否 | 是否主映射 |
| note | text | 是 | 备注 |
| created_at | timestamptz | 否 | 创建时间 |

约束建议：

### 11.14 小结

这 8 张核心表和 4 张关联表构成第一版数据库落地骨架：

- Breeding 主体层：
  - `brd_program`
  - `brd_material`
  - `brd_trial`
  - `brd_plot`
  - `brd_observation`
  - `brd_biosample`
  - `brd_assay`
  - `brd_data_file`
- 关联层：
  - `brd_dataset_subject_link`
  - `brd_dataset_assay_link`
  - `brd_variant_sample_map`
  - `brd_phenotype_subject_map`

其中最重要的控制原则不变：

- Dataset 域仍然先独立成立
- Breeding 域是结构化解释层
- 关联层允许后补，不阻塞资产先入库

### 11.15 关键约束与删除策略建议

为了让这批表后续能真正落成 PostgreSQL DDL，本节补充第一版统一约束口径。

外键删除策略建议：

- `brd_program -> brd_material / brd_trial`
  - 建议 `on delete restrict`
- `brd_trial -> brd_plot`
  - 建议 `on delete restrict`
- `brd_material -> brd_plot / brd_biosample`
  - 建议 `on delete restrict`
- `brd_plot -> brd_observation`
  - 建议 `on delete set null`
- `brd_material -> brd_observation`
  - 建议 `on delete set null`
- `brd_biosample -> brd_assay`
  - 建议 `on delete restrict`
- `brd_assay -> brd_data_file`
  - 建议 `on delete restrict`
- 关联层指向 breeding 主体的可空外键
  - 建议 `on delete set null`
- 关联层指向 dataset/version/asset 的外键
  - 建议与 Dataset 域主线保持一致，默认 `on delete restrict`

原因：

- breeding 主体层是业务主数据，不适合级联物理删除
- `observation` 和关联表更像事实记录与桥表，允许在主体删除受限时保留痕迹
- Dataset 域已经是文件真相域，不能被关联层反向拖着级联删除

推荐检查约束：

- `brd_observation`
  - `observation_level = 'plot'` 时，`plot_id` 必填
  - `observation_level = 'material'` 时，`material_id` 必填
  - 数值、文本、评分三类值至少允许一类非空
- `brd_data_file`
  - `source_mode in ('dataset_file', 'external_uri')`
  - `source_mode = 'dataset_file'` 时，`asset_file_id` 必填
  - `source_mode = 'external_uri'` 时，`uri_snapshot` 必填
- `brd_dataset_subject_link`
  - `program_id/material_id/trial_id/plot_id/biosample_id` 必须且只能有一列非空
- `brd_variant_sample_map`
  - 保持“先抽样本名、后补映射”，因此不要求主体列必填
- `brd_phenotype_subject_map`
  - 保持“先登记 row、后补主体映射”，因此不要求 `trial_id/plot_id/material_id` 立即齐全

### 11.16 暂缓规范化对象

为了先把第一版骨架落稳，本轮明确暂缓以下对象的实体化：

- `brd_trial_entry`
- `brd_trait`
- `brd_protocol`
- `brd_plot_plant`
- `brd_block`
- `brd_treatment`

这并不表示这些对象不重要，而是当前优先级更适合先落：

- 可注册文件资产
- 可表达 breeding 主体
- 可建立 dataset 与主体/assay 的桥

后续当以下需求变强时，再进入下一轮规范化：

- 复杂田间设计和参试材料管理
- 性状字典统一治理
- 观测协议版本化
- 单株级或小区内分层观测
- 更严格的试验随机化与处理模型

## 12. 当前旧模型的重新定位

### 12.1 project

当前系统中的 `project` 更接近：

- 平台级项目容器
- 兼容层业务项目

它不应直接等价为 breeding 中的 `Program`，但两者在过渡期可建立映射或兼容。

### 12.2 sample

当前系统中的 `sample` 更像旧式通用样本。

后续建议：

- 在业务语义上向 `BioSample` 收敛
- 但不要把它继续当作所有数据的顶层主对象

### 12.3 experiment

当前系统中的 `experiment` 更像 SRA 风格实验对象。

后续建议：

- 在平台语义上向 `Assay` 收敛
- 继续兼容旧接口，但不再作为平台总模型的中心

### 12.4 文件管理主线

当前已经形成的 `dataset/version/asset/file/lineage` 主线不应被这次设计推翻。

相反，它应成为两个域之间稳定的中间底座。

## 15. 最小可执行结论

如果要把这版设计压缩成一句最重要的规则，就是：

**任何数据文件都应先能独立注册为 Dataset 资产；结构化育种/组学元数据是增强层，不是前置门槛。**

这条规则落到平台上就是：

- Dataset 域是基础域
- Breeding 域是解释域
- 关联层是桥梁

而不是：

- 没有 `Program / Material / BioSample / Assay` 就不能注册数据

---

## 16. 最终结论

当前平台后续设计应正式收敛为：

### A. Dataset 资产域

- 独立成立
- 先落文件
- 先管版本
- 先管资产
- 先管查询

### B. Breeding 业务域

- 以 `Program / Material / Trial / Plot / Observation / BioSample / Assay` 为主线
- 用于解释数据的业务意义

### C. 关联层

- 允许后补
- 允许延迟绑定
- 不阻塞资产先入库

一句话概括：

Dataset 平台负责“数据文件先落下来”，
Breeding 平台负责“这些数据在育种业务里意味着什么”，
两者通过关联层松耦合，而不是在注册入口强耦合。
