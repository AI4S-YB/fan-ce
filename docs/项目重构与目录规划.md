# 项目重构与目录规划

## 1. 目标

将当前仓库从“仅包含后台后端代码”的状态，规划为一个同时容纳后台系统和前台系统的统一项目目录。

重构后的产品形态如下：

- 后台系统
  - 后端：负责数据管理、权限、导入、审核、发布、前台 API 支撑
  - 前端：负责内部管理页面
- 前台系统
  - 前端：负责组学数据库网站展示页面
  - 不单独建设后端，直接调用后台后端提供的公开 API

## 2. 当前现状

当前仓库实际只包含一套后端代码，主要职责是：

- 管理组学数据及其元数据
- 提供样本、实验、基因集等后台管理接口
- 提供组学数据查询、展示、下载能力
- 通过 API 支撑前台组学数据库网站

也就是说，当前仓库中的代码本质上是：

> 后台系统的后端，同时也是前台系统的数据服务层。

## 3. 规划原则

本次先做“大规划”和“目录骨架建立”，随后已完成后端相关代码的第一轮归位。

原始规划原因：

1. 当前后端仍可继续运行，不因为目录移动导致配置、导入路径、启动方式整体失效
2. 后台前端和前台前端可以先作为空目录保留，等待后续拷贝旧代码
3. 后续可以按模块逐步迁移，而不是一次性重构整个仓库

## 4. 目标目录结构

建议目标结构如下：

```text
fan-ce/
  backend/
    api-server/              # 后台后端 FastAPI 主服务
    sdk/                     # Python SDK
    mcp/                     # MCP 服务

  frontend/
    admin-web/               # 后台前端，当前先保留空目录
    portal-web/              # 前台网站前端，当前先保留空目录

  docs/                      # 设计文档、重构文档
```

## 5. 各目录职责

### 5.1 `backend/api-server`

当前已经放置并运行当前这套后端代码。

职责包括：

- 后台管理 API
- 前台公开查询 API
- 组学数据查询与文件解析能力
- 统一认证、权限、数据发布控制
- 外部系统集成

### 5.2 `frontend/admin-web`

后台管理前端。

职责包括：

- 登录与权限控制
- 数据导入与维护
- 样本、实验、数据库、基因集管理
- 项目、团队、用户、角色管理
- 数据发布和运营管理

### 5.3 `frontend/portal-web`

前台组学数据库网站前端。

职责包括：

- 数据浏览
- 检索与筛选
- 详情展示
- 下载入口
- 图表或可视化页面

注意：

- 该系统只有前端页面
- 不单独建设后端
- 所有数据能力由 `backend/api-server` 提供

## 6. 后端内部逻辑建议

未来即使仍保留单体后端，也建议在逻辑上分成三层接口域：

- `admin`
  - 后台管理接口
  - 需要登录和权限控制
- `public`
  - 前台网站使用的公开只读接口
  - 只返回允许公开的数据
- `omics`
  - 真正的组学数据查询与文件处理能力
  - 为 `admin` 和 `public` 共同复用

建议 API 形态：

```text
/api/admin/...
/api/public/...
/api/omics/...
```

## 7. 重构阶段建议

建议分四阶段进行：

### 第一阶段：目录规划

- 建立 `backend/`、`frontend/` 目录
- 建立 `frontend/admin-web`、`frontend/portal-web` 空目录
- 输出本规划文档

### 第二阶段：后端归位

- 将当前根目录后端代码逐步迁移到 `backend/api-server`
- 修正启动命令、配置路径、构建路径
- 保证迁移后服务仍可独立运行

### 第三阶段：前端归位

- 将旧的后台前端拷贝到 `frontend/admin-web`
- 将旧的前台网站代码拷贝到 `frontend/portal-web`
- 统一 API 地址和环境配置

### 第四阶段：接口收敛

- 明确后台接口和前台接口边界
- 增加发布态控制
- 限制前台只读能力
- 去掉前台对后台管理接口的直接依赖

## 8. 当前处理结果

当前已经完成以下事项：

1. 建立目标目录骨架
2. 预留两个前端空目录
3. 将后端主服务收敛到 `backend/api-server`
4. 将 Python SDK 收敛到 `backend/sdk`
5. 将 MCP 服务收敛到 `backend/mcp`
6. 删除旧 `.git` 信息，作为新项目重新开始

## 9. 后续建议

下一步最合适的工作是继续完善“系统分层设计”，明确：

- 哪些接口属于后台管理
- 哪些接口属于前台展示
- 哪些能力属于共用组学能力层
- 数据从后台导入到前台展示的完整链路

这一步做清楚之后，再进行后端内部模块重构会更稳。
