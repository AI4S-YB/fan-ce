# 种质资源导入与浏览设计草案

## 1. 文档目标

本文档用于定义当前平台中 `种质资源` 模块的下一阶段建设方式，重点解决以下问题：

- 用户如何上传符合规范的种质资源 `xls/xlsx`
- 系统如何在导入前校验文件格式是否符合要求
- 种质资源如何绑定到一个明确的 `NCBI Taxonomy` 锚点
- 导入完成后，如何形成可筛选、可搜索、可继续被 breeding 使用的种质列表
- 现有“知识图谱/关系浏览”能力如何与新的导入模型整合

这份设计优先服务当前月季场景，但必须保留多物种扩展能力。

---

## 2. 现状判断

当前代码中的种质资源模块，本质上还是“单个 Excel 文件驱动的浏览器”：

- 前端列表页直接依赖固定文件路径
- 后端通过 `xlsx -> graph.pkl` 生成关系图
- 查询、详情、统计、关系、图谱都是围绕这个文件路径运行
- 还没有真正的平台化导入入口
- 还没有导入前格式校验
- 还没有物种锚点
- 还没有按物种过滤

当前月季样例文件真实表头如下：

| 列名 | 含义 |
|---|---|
| `ID` | 种质编号 |
| `chinese_name` | 中文名称 |
| `english_name` | 英文名称 |
| `P` | 父本编号 |
| `M` | 母本编号 |
| `P_name` | 父本名称 |
| `M_name` | 母本名称 |
| `育种历史` | 自由文本历史 |
| `花朵性状` | 自由文本花部描述 |
| `植株特征` | 自由文本植株描述 |
| `用途` | 用途描述 |

这说明当前文件不是“任意 Excel”，而是一个带谱系关系的种质主表。

因此下一阶段的设计不应该再继续强化“页面直接查文件路径”，而应该转成：

`模板化导入 -> 结构化落库 -> 关系衍生 -> 页面浏览`

---

## 3. 设计目标

### 3.1 业务目标

- 支持用户导入种质资源 Excel
- 导入前能明确告诉用户“哪些列不对、哪一行有问题、为什么不能导入”
- 导入时必须绑定一个物种 taxonomy 锚点
- 导入后形成可筛选、可搜索、可详情查看的种质列表
- 导入后的种质资源可以继续作为 breeding 的材料来源

### 3.2 技术目标

- 不再让种质资源浏览依赖固定本地文件路径
- Excel 保留为原始来源文件，但不是运行态唯一真相
- PostgreSQL 作为种质资源主存储
- 谱系图和知识图谱作为衍生视图，不作为唯一数据来源
- taxonomy 采用“本地缓存优先，必要时远端补齐”的策略

---

## 4. 总体设计

### 4.1 核心原则

种质资源模块分三层：

1. 导入层
   - 处理上传、模板校验、预览、导入确认
2. 主数据层
   - 把种质资源和谱系关系落到 PostgreSQL
3. 展示层
   - 列表、详情、知识图谱、被 breeding 引用情况

### 4.2 推荐的数据流

```text
用户上传 xlsx
    ->
选择 taxonomy 锚点
    ->
模板校验
    ->
预览导入结果
    ->
确认导入
    ->
写入 PostgreSQL 主表
    ->
生成谱系关系表 / 图谱缓存
    ->
列表与详情页可浏览
    ->
后续 breeding 项目引用这些种质
```

## 5. 物种与 taxonomy 设计

## 5.1 为什么必须有 taxonomy 锚点

如果平台未来支持多个物种，种质资源不能只靠“中文名”或“项目名”区分。

必须有一个稳定锚点，用于：

- 物种过滤
- 和基因组、表型、变异等数据建立跨模块关联
- 后续 breeding 项目下材料与组学数据的生物学一致性判断
- 防止不同物种使用同样 accession 样式而混淆

## 5.2 推荐策略

推荐采用：

- 导入批次级别强制选择一个 `NCBI taxonomy anchor`
- 第一阶段一个导入文件只绑定一个物种锚点
- 不在第一阶段支持“一个 Excel 混多个物种”

这对当前月季数据最合适，也能显著简化前端和校验逻辑。

## 5.3 taxonomy 数据来源策略

推荐分两步：

### V1

- 前端提供 taxonomy 搜索框
- 后端先查本地缓存表
- 本地未命中时，再走远端 NCBI 查询或人工补录缓存
- 用户最终选定一个 taxonomy 记录用于本次导入

### V2

- 后台支持完整同步 NCBI taxonomy 到本地
- 前端只查本地 taxonomy 索引
- 远端只用于周期性更新，不直接参与页面请求

## 5.4 taxonomy 记录建议字段

- `tax_id`
- `scientific_name`
- `common_name`
- `rank`
- `parent_tax_id`
- `lineage`
- `lineage_names`
- `is_local_cached`
- `last_sync_time`

结论：

当前已有的 organism/taxonomy 接口只是内存示例，不能直接承担这项功能，需要重做成本地缓存模型。

## 6. Excel 模板设计

## 6.1 模板策略

不要把“所有物种都用一套完全一样的 Excel 模板”作为强约束。

更合理的是：

- 平台定义一个“种质核心字段集合”
- 每个模板 profile 再定义自己的扩展列

这样既能支持月季，又不会把未来别的物种卡死。

## 6.2 核心字段

建议每个种质模板都至少映射到以下核心字段：

- `accession_id`
- `display_name`
- `scientific_name_optional`
- `father_accession_optional`
- `mother_accession_optional`
- `origin_optional`
- `remark_optional`

## 6.3 月季模板 v1

月季当前样例可以定义为一个模板 profile，例如：

- 模板名：`rose_germplasm_v1`
- 必填列：
  - `ID`
  - `chinese_name`
- 推荐列：
  - `english_name`
  - `P`
  - `M`
  - `P_name`
  - `M_name`
  - `育种历史`
  - `花朵性状`
  - `植株特征`
  - `用途`

## 6.4 校验规则

导入前校验分四层：

### A. 文件级校验

- 是否为 `xls/xlsx`
- 是否只有一个主工作表，或是否找到指定工作表
- 是否能正常读取表头

### B. 模板级校验

- 必填列是否存在
- 列名是否完全匹配或在允许别名范围内
- 是否存在重复列

### C. 行级校验

- `ID/accession` 是否为空
- accession 是否重复
- 父本/母本编号是否引用非法值
- 文本长度是否超限
- 空白行是否应跳过

### D. 关系级校验

- 父本或母本是否引用自身
- 父母本编号是否在当前文件中存在，或者是否允许外部未收录亲本
- 是否形成明显异常关系

## 6.5 校验结果展示

前端不能只弹一句“格式错误”。

必须返回结构化报告：

- 文件是否通过
- 模板 profile
- 总行数
- 有效行数
- 错误数
- 警告数
- 错误明细表
  - 行号
  - 列名
  - 错误类型
  - 错误说明
- 建议修复动作

## 7. PostgreSQL 主数据设计

## 7.1 为什么不能继续只查 Excel

继续直接查 Excel 的问题很明显：

- 不利于分页、过滤、统计
- 不利于多用户访问
- 不利于与 breeding 引用做稳定关联
- 不利于按物种筛选
- 不利于后续扩展更多字段和标签

因此推荐：

- Excel 作为原始导入文件保留
- PostgreSQL 作为种质资源主数据存储
- 图谱/关系缓存作为衍生产物

## 7.2 建议的主表

### `germplasm_import_batch`

用于记录一次导入任务：

- `id`
- `batch_code`
- `template_profile`
- `source_filename`
- `source_file_path`
- `taxonomy_tax_id`
- `taxonomy_scientific_name_snapshot`
- `status`
- `total_rows`
- `valid_rows`
- `error_rows`
- `warning_rows`
- `imported_by`
- `import_time`
- `validation_report_json`

### `germplasm_record`

用于存实际种质记录：

- `id`
- `batch_id`
- `taxonomy_tax_id`
- `accession_id`
- `display_name`
- `scientific_name_snapshot`
- `common_name_snapshot`
- `father_accession`
- `mother_accession`
- `father_name_snapshot`
- `mother_name_snapshot`
- `source_row_no`
- `status`
- `attributes_json`
- `search_text`
- `created_at`
- `updated_at`

说明：

- 月季模板中的 `育种历史/花朵性状/植株特征/用途` 建议先落到 `attributes_json`
- 等后续真正做育种性状结构化时，再拆更细字段

### `germplasm_lineage_edge`

用于保存谱系关系：

- `id`
- `taxonomy_tax_id`
- `child_accession`
- `parent_accession`
- `parent_role`
  - `father`
  - `mother`
- `batch_id`
- `source_row_no`

## 7.3 图谱缓存的定位

现有 `graph.pkl` 不建议继续作为主事实源。

建议改成：

- 从 `germplasm_record + germplasm_lineage_edge` 衍生图谱数据
- 可以继续生成缓存文件
- 也可以直接查询 PG 后组装图数据

结论：

图谱是展示层缓存，不是主数据。

## 8. 页面设计

## 8.1 菜单结构

当前保持：

- 一级菜单：`种质资源`
- 二级菜单：`种质列表`

建议补充：

- 二级菜单：`导入种质资源`
- 二级菜单：`导入记录`

如果你希望更简洁，也可以只保留一个列表页，把“导入”和“记录”做成弹窗或抽屉入口。

但从长期可维护性看，我更推荐单独页面。

---

## 8.2 页面一：种质列表

这是用户的主工作页。

### 页面目标

- 浏览已导入种质
- 先按物种过滤，再按关键词查找
- 查看详情、谱系、breeding 引用

### 顶部筛选区

- `物种选择`
  - 从 taxonomy 缓存中选择
  - 支持搜索 scientific name/common name
- `关键词`
  - 搜 accession
  - 搜中文名
  - 搜英文名
  - 搜部分属性文本
- `导入批次`
  - 可选
- `状态`
  - 正常 / 警告 / 已停用

### 操作区

- `导入种质资源`
- `下载模板`
- `导出当前筛选`
- `查看图谱`

### 列表字段建议

- `accession`
- `显示名称`
- `学名`
- `taxonomy`
- `父本`
- `母本`
- `breeding 引用数`
- `导入批次`
- `更新时间`
- `操作`

### 查询顺序

用户体验上，按你的要求建议明确为：

1. 先选择 `物种`
2. 再输入 `关键词`

也就是说：

- 物种是一级过滤器
- 关键词是在物种过滤结果内继续搜索

---

## 8.3 页面二：导入种质资源

这是新增页面，建议做成分步向导。

### Step 1 选择导入配置

- 选择模板 profile
  - 先至少支持 `rose_germplasm_v1`
- 选择 taxonomy 锚点
  - 搜索 NCBI taxonomy
  - 确认 scientific name / rank / lineage
- 下载模板

### Step 2 上传文件

- 上传 `xls/xlsx`
- 显示文件名、大小、上传时间

### Step 3 校验预览

- 显示校验结果总览
- 展示错误行明细
- 展示警告行明细
- 预览前 N 条导入记录

### Step 4 导入确认

- 确认写入数据库
- 导入成功后跳转到“种质列表”
- 自动带上当前 taxonomy 过滤条件

## 9. 接口设计建议

## 9.1 taxonomy

- `GET /taxonomy/{tax_id}`
- `POST /taxonomy/search`
- `POST /taxonomy/cache/sync`

说明：

- 当前内存版接口不能直接用
- 需要替换成真实缓存查询

## 9.2 germplasm 导入

- `POST /breeding/germplasm/import/validate`
  - 上传文件并返回校验报告，不写库
- `POST /breeding/germplasm/import/commit`
  - 根据校验通过的临时结果正式导入
- `POST /breeding/germplasm/import/batches`
  - 导入记录列表
- `POST /breeding/germplasm/import/batch-info`
  - 单批次详情和校验报告

## 9.3 germplasm 浏览

- `POST /breeding/germplasm/list`
  - 支持 `taxonomy_tax_id + keyword + batch_id + status`
- `POST /breeding/germplasm/info`
  - 按 `accession_id + taxonomy_tax_id` 返回种质详情、lineage summary 和 breeding usage
- `POST /breeding/germplasm/statistics`
  - 返回 PG 谱系统计
- `POST /breeding/germplasm/relationship`
  - 返回两个 accession 的关系判断
- `POST /breeding/germplasm/relationships/batch`
  - 根据选中节点返回图谱

## 10. 与 breeding 的关系

这部分很关键。

种质资源必须先独立成立，不应该要求“先建 breeding 项目才能导入种质”。

正确顺序应当是：

1. 先导入种质资源
2. 形成稳定 accession 列表
3. breeding 项目后续从种质资源中选材料

也就是说：

- `种质资源` 是 breeding 的上游资源池
- `breeding material` 是项目上下文中的使用视图
- 两者不是同一个对象

这和你现在想法是一致的。

---

## 11. 第一阶段实施建议

建议先做一个明确的 Phase 1，不要一次做太满。

### Phase 1

- 支持 `rose_germplasm_v1` 模板导入
- taxonomy 锚点选择
- 上传前校验
- 校验报告页面
- 导入到 PostgreSQL
- 种质列表支持
  - 物种过滤
  - 关键词搜索
- 详情页展示来源批次和 taxonomy

### Phase 2

- 从 PG 生成图谱，不再依赖固定文件路径
- 导入记录页
- 图谱页按筛选条件加载
- 与 breeding material 反向引用整合

### Phase 3

- 多模板 profile
- 完整 taxonomy 本地同步
- 更复杂的谱系分析
- 属性结构化拆分

---

## 12. 最终建议

当前最合理的主线不是继续增强“固定 Excel 文件浏览”，而是：

```text
种质 Excel 模板化
    ->
taxonomy 锚点选择
    ->
导入前校验
    ->
PostgreSQL 主数据落库
    ->
列表/详情/图谱/ breeding 引用统一浏览
```

一句话收束：

`种质资源` 应该从“文件驱动查询页”升级成“可导入、可校验、可筛选、可被 breeding 引用的主数据模块”。
